# itinerry Visa Assessment — UX/UI Conversion Spec
### Google Form → Mobile Web App

---

## 1. บริบทที่กำหนดทุกการตัดสินใจ

| แกน | ค่า | สิ่งที่ตามมา |
|---|---|---|
| **ความวิตกกังวล (Anxiety)** | **สูง** — วีซ่า = เรื่องใหญ่ ผิดพลาดไม่ได้ | ต้อง reassure ตลอด ลดความกลัว "ตอบผิด" |
| **ความรู้ในเรื่อง (Domain knowledge)** | **ต่ำ** — ผู้ยื่นส่วนใหญ่ทำครั้งแรก ไม่รู้ว่าเอกสารแต่ละอย่างคืออะไร | ต้องมี help text / ตัวอย่าง / คำอธิบายทุกจุดที่ใช้ศัพท์เฉพาะ |
| **แพลตฟอร์ม** | **Mobile-first**, LINE เป็นช่องทางหลัก | thumb zone, คอลัมน์เดียว, native input, ปุ่มใหญ่ |
| **ผู้ใช้** | คนไทย | ภาษาเรียบง่าย เป็นกันเอง ไม่ราชการ |

> **หัวใจของงานนี้:** ผู้ยื่นวีซ่ากลัวว่าการ "ยอมรับ" บางอย่าง (เคยถูกปฏิเสธ, overstay, เงินน้อย, ไม่มีเอกสาร) จะทำให้เสียโอกาส — แต่ข้อมูลจริงคือสิ่งที่ทีมต้องใช้พอดี ดังนั้นงานออกแบบ #1 คือ **ทำให้คนกล้าตอบตามจริงโดยไม่รู้สึกถูกตัดสิน**

## 2. หลักการใหญ่ 7 ข้อ (ใช้กับทั้งฟอร์ม)

1. **ทีละคำถาม (one question at a time)** — เลิก scroll ยาวแบบ Google Form เปลี่ยนเป็นทีละหน้าจอ ลด cognitive load (Sweller) และลดความกังวลว่า "จะยาวแค่ไหน" — งานวิจัยพบ completion สูงกว่าฟอร์มหน้าเดียว ~25%
2. **Branching = ซ่อนคำถามที่ไม่เกี่ยวข้องไปเลย** — ฟอร์มมี skip logic อยู่แล้ว ใน web app ผู้ใช้จะ *ไม่เห็น* คำถามที่ไม่ตรงกับตัวเอง ทำให้ "รู้สึกสั้นลง" (personalization)
3. **Progress แบบ section ไม่ใช่ %** — เพราะ branching ทำให้จำนวนคำถามไม่เท่ากัน การโชว์ % จะกระตุกและหลอก ให้ใช้ "ขั้นที่ 3 จาก 6 · อาชีพ" แทน (chunking) และทำให้ช่วงต้นรู้สึกเร็ว (endowed progress / goal gradient)
4. **Native mobile input** — date picker, แป้นตัวเลขสำหรับเบอร์โทร, แป้นอีเมล, tap target ใหญ่ ปุ่มหลักอยู่โซนนิ้วโป้ง
5. **ลดความกลัวที่คำถาม sensitive** — ใส่ micro-copy ให้กำลังใจตรงคำถามเรื่องการถูกปฏิเสธ/overstay/การเงิน
6. **Plain-language help (เพราะ knowledge ต่ำ)** — ทุกศัพท์เฉพาะ (Invitation Letter, DBD, Acceptance Letter, Ties to home) ต้องมีคำอธิบายสั้นหรือ tooltip
7. **Peak-end rule** — เปิดด้วย welcome ที่อุ่นใจ ปิดด้วยหน้า confirmation ที่ชัดเจน (ความรู้สึกตอนเปิด+ปิดกำหนดความทรงจำต่อทั้งแบรนด์)

## 3. โครงสร้าง flow ใหม่ (ภาพรวม screen-by-screen)

```
[0] Welcome screen (ใหม่)    ── value + trust + "~2 นาที" + privacy 1 บรรทัด
[1] Consent (ยุบ)            ── 1 toggle, PDPA เต็มซ่อนใน bottom-sheet
[2] ปลายทาง + ประเภทวีซ่า     ── สัญชาติ + การ์ดเลือกประเภท = จุดแยก branch
[3] คำถามเฉพาะประเภทวีซ่า     ── (เห็นเฉพาะที่เกี่ยวกับตัวเอง)
[4] อาชีพ + เอกสารการเงิน     ── (branch ตามอาชีพ)
[5] คัดกรองหลัก (sensitive)   ── ถูกปฏิเสธ / overstay / เงินออม / ผูกพันไทย
[6] "เกือบเสร็จ!" ข้อมูลติดต่อ  ── ชื่อ → เบอร์ → อีเมล + ช่องทาง (LINE) + ช่วงเวลา
[7] ปิดท้าย                  ── รู้จักจากช่องไหน + รับรองข้อมูลจริง → submit
[8] Confirmation (ใหม่)       ── ยืนยัน + ติดต่อกลับเมื่อไหร่ + ปุ่ม Add LINE
```

**ย้ายข้อมูลติดต่อ (ชื่อ/เบอร์/อีเมล/รู้จักจากช่องไหน) + รับรองข้อมูลจริง ไปไว้ชุดท้าย — เป็นการตัดสินใจที่มี tradeoff ชัดเจน:**

- ✅ **เหตุผลที่ทำให้ดีในบริบทนี้:** เปลี่ยน commitment จาก "ยอมให้ข้อมูลก่อน" เป็น "ลงแรงทำ assessment ก่อน" (IKEA effect / sunk cost) — พอตอบคำถามวีซ่าครบแล้วคนจะอยาก "รับผลประเมิน" จึงให้เบอร์/อีเมลโดยไม่รู้สึกฝืน; ลดความรู้สึก "กับดักเก็บ lead" ตอนต้นที่ทำให้คนกังวล (โดยเฉพาะคนไทยที่ระวังเรื่องเบอร์โทร/สายขายของ); และตอนท้ายคำถาม "ขออีเมลทำไม?" มีคำตอบชัด = "ส่งผลประเมินให้คุณ"
- ⚠️ **ความเสี่ยงที่ต้องรับ:** ถ้าคนทำ assessment จนจบแต่หนีตอนขั้นกรอกติดต่อ = เราได้คำตอบแต่ **ติดต่อกลับไม่ได้เลย** (เทียบกับถ้าถามต้น อย่างน้อยได้ lead ไว้ตาม) — แลกกับที่ของเดิมเคยอ้าง "ถามอีเมลก่อน response +9%" ซึ่งใช้กับ survey สั้นๆ ไม่ใช่ assessment ยาวที่ commitment มาจากความพยายาม
- 🛡️ **วิธีลดความเสี่ยง:** (1) ข้อความเปลี่ยนผ่านก่อนถึงชุดติดต่อให้กรอบเป็นการ "ส่งมอบคุณค่า" — "เกือบเสร็จแล้ว! บอกเราหน่อยว่าจะส่งผลประเมินให้คุณยังไง" (2) ถ้าเครื่องมือรองรับ ให้บันทึกคำตอบแบบ partial ไว้ แม้คนไม่กรอกติดต่อ จะได้ข้อมูลเชิงรวม (3) ขั้นติดต่อสั้นและคุณค่าชัด

## 4. Mapping รายคำถาม → UI Pattern

### Consent & กฎหมาย (หน้าแรกของ Google Form)

| คำถาม (ปัจจุบัน) | UI ที่แนะนำบนมือถือ | หลักการ / เหตุผล |
|---|---|---|
| PDPA consent (Q1) + รับทราบ (Q2) | **Welcome screen + 1 หน้า consent**: นโยบายเต็มซ่อนหลังลิงก์ "อ่านนโยบาย" (เปิดเป็น bottom-sheet) เหลือ toggle เดียวให้กด | Processing fluency + อย่าเปิดด้วยกำแพงตัวหนังสือ (หน้าแรกคือจุดที่คนตัดสินใจว่าจะไปต่อไหม) — *เช็คเรื่องการแยก consent ตามกฎหมายกับทีมกฎหมาย* |
| รับรองว่าข้อมูลจริง (certification) | **ย้ายไปไว้ขั้นสุดท้าย** ก่อนปุ่ม submit เป็น checkbox attestation | ตำแหน่งตามธรรมชาติ (รับรองหลังกรอกเสร็จ) ลด friction ตอนต้น |

### S1 · ปลายทาง + ประเภทวีซ่า

| คำถาม (ปัจจุบัน) | UI ที่แนะนำบนมือถือ | หลักการ / เหตุผล |
|---|---|---|
| สัญชาติ (Q4, radio Thai/Other) | ปุ่มใหญ่ 2 ตัว (segmented), **default = ไทย**; เลือก "Other" แล้วค่อยโผล่ช่องค้นหาประเทศ | majority default ลดการกด; progressive disclosure — *สัญชาติเป็นข้อมูลประเมิน (พาสปอร์ตกำหนดเงื่อนไขวีซ่า) จึงอยู่ต้น ไม่ได้ย้ายไปท้าย* |
| ประเทศปลายทาง (Q8, dropdown 16) | **searchable select + ธงชาติ**, ดันตัวเลือกยอดฮิตขึ้นบน | >6 ตัวเลือกต้องมีค้นหา (เกิน Hick's Law); ธง = จำง่าย |
| ประเภทวีซ่า (Q9, radio 4 → branch) | **การ์ดใหญ่มีภาพ**: ไอคอน + ป้าย + คำอธิบาย 1 บรรทัด (เช่น "ท่องเที่ยว — ไปเที่ยวเป็นหลัก") | จุดแยก branch สำคัญ; การ์ดมี helper ช่วยคน knowledge ต่ำเลือกถูก |

> ℹ️ **ชื่อ-นามสกุล (Q3) · เบอร์โทร (Q5) · อีเมล (Q6) · รู้จักจากช่องไหน (Q7)** ย้ายไปอยู่ "ชุดสุดท้าย" แล้ว ดูหัวข้อด้านล่าง

### วันที่ทั้งหมด (Q10, 11, 13, 17, 18, 21)

| คำถาม (ปัจจุบัน) | UI ที่แนะนำบนมือถือ | หลักการ / เหตุผล |
|---|---|---|
| วันเดินทาง/วันกลับ (free text date) | **date picker ปฏิทิน**; คู่ไป-กลับใช้ **range picker** หน้าเดียว (โชว์จำนวนวันให้อัตโนมัติ) | native input; range picker ดูโปร; validate วันกลับ > วันไป |
| วันเริ่มเรียน (Q21) | date picker เดี่ยว | เหมือนกัน |

> ⚠️ อย่าใช้ text ว่างให้พิมพ์วันที่เด็ดขาด — เป็นบ่อเกิดของ error และความหงุดหงิดบนมือถือ

### คำถามเฉพาะ branch (Visitor / Business / Student)

| คำถาม (ปัจจุบัน) | UI ที่แนะนำบนมือถือ | หลักการ / เหตุผล |
|---|---|---|
| เคยได้วีซ่า UK/Schengen/... ใน 5 ปี (Q12, Q20, check all) | **multi-select ชิป + ธง**; "ไม่เคย" เป็นตัวเลือกที่กดแล้วล้างตัวอื่น; **กรอบเชิงบวก** "เลือกที่เคยได้ — ยิ่งมียิ่งดีต่อเคส" | เปลี่ยนคำถามให้เป็น "สัญญาณบวก" สร้างความมั่นใจ; logic ตัวเลือก exclusive |
| สถานะคนที่ไปเยี่ยม (Q14, radio 4) | single-select list + คำอธิบายสั้นแต่ละตัวเลือก | knowledge ต่ำ ต้องช่วยให้เข้าใจ |
| ความสัมพันธ์กับผู้เชิญ (Q15, radio 5) | single-select มีไอคอน; คงการแยก "คู่สมรส (จดทะเบียน)" vs "แฟน" ให้ชัด | ความต่างนี้มีผลต่อวีซ่าจริง |
| เอกสารที่ผู้เชิญส่งให้ได้ (Q16, check all) | **checklist multi-select + tooltip (i)** อธิบายเอกสารแต่ละชนิด | low knowledge: ต้องบอกว่าแต่ละเอกสารคืออะไร |
| มี Invitation/Acceptance Letter ไหม (Q19, Q22, radio) | segmented 3 ตัวเลือก (มีแล้ว/กำลังขอ/ไม่มี) | toggle เร็ว |

### S3–S4 · อาชีพ + เอกสารการเงิน

| คำถาม (ปัจจุบัน) | UI ที่แนะนำบนมือถือ | หลักการ / เหตุผล |
|---|---|---|
| อาชีพปัจจุบัน (Q24, radio 7 → branch) | single-select list **มีไอคอน** แต่ละอาชีพ | 7 ตัวเลือกใช้ list ได้; ไอคอนช่วยกวาดสายตา |
| เอกสารงาน/รายได้/ภาษี/DBD (Q25,26,27,28, radio) | **segmented 3 ตัวเลือก** (มีครบ/มีบางส่วน/ยังไม่มี) — **สีเป็นกลาง** อย่าทำให้ "ยังไม่มี" ดูเหมือนสอบตก | toggle เร็ว; หลีกเลี่ยง framing ที่เพิ่มความกังวล |
| ใครรับผิดชอบค่าใช้จ่าย (Q23, Q29, radio) | single-select ปุ่ม | ตรงไปตรงมา |

### S5 · คัดกรองหลัก (โซน sensitive — ออกแบบให้ดีที่สุด)

| คำถาม (ปัจจุบัน) | UI ที่แนะนำบนมือถือ | หลักการ / เหตุผล |
|---|---|---|
| เคยถูกปฏิเสธวีซ่า (Q30, radio Yes/No) | toggle Yes/No + **micro-copy ให้กำลังใจ** "ตอบตามจริงช่วยให้เราเตรียมเคสได้แม่นขึ้น — เราไม่ตัดสิน"; ตอบ "เคย" → ค่อยโผล่ Q31 | จุดที่ความกังวลพีค ต้องลดความกลัวการยอมรับ |
| ระบุประเทศ/ปีที่ถูกปฏิเสธ (Q31, text) | conditional: **dropdown ประเทศ + ปี picker** (ไม่ใช่ text ว่าง) | ได้ข้อมูลสะอาด มีโครงสร้าง |
| เคย overstay (Q32, radio Yes/No) | เหมือน Q30 + reassurance; ตอบ "เคย" → โผล่ Q33 | เหมือนกัน |
| ระบุรายละเอียด overstay (Q33, text) | conditional structured input | เหมือนกัน |
| ยอดเงินออม (Q34, radio 4 ช่วง) | single-select **ปุ่มช่วงเงิน** + micro-copy ความเป็นส่วนตัว "ข้อมูลนี้เก็บเป็นความลับ" | ช่วงเงินรุกล้ำน้อยกว่าจำนวนเป๊ะ; trust ตรงจุด |
| ความผูกพันกับไทย (Q35, check all 6) | **checklist multi-select** + **กรอบเชิงบวก** "เลือกทุกข้อที่ใช่ — ยิ่งมากยิ่งช่วยเคส"; "ไม่มีข้อใด" = exclusive | เปลี่ยนเป็นคำถามสร้างความมั่นใจ; logic exclusive |

### 🏁 ชุดสุดท้าย · ข้อมูลติดต่อ + นัดติดต่อกลับ + รับรอง

> **ข้อความเปลี่ยนผ่าน** (แสดงก่อนเข้าชุดนี้): *"เกือบเสร็จแล้ว! 🎉 เหลือแค่บอกเราว่าจะส่งผลประเมินให้คุณยังไง"* — กรอบการขอข้อมูลติดต่อให้เป็น "ส่งมอบผลให้คุณ" ไม่ใช่ "เก็บข้อมูลเรา"

| คำถาม (ปัจจุบัน) | UI ที่แนะนำบนมือถือ | หลักการ / เหตุผล |
|---|---|---|
| ชื่อ-นามสกุล (Q3, text) | text บรรทัดเดียว, auto-cap, auto-advance | สั้นเร็ว; ขอหลังผู้ใช้ลงแรงทำ assessment แล้ว (sunk cost) |
| เบอร์โทร (Q5, text) | แป้นตัวเลข (tel), input mask เบอร์ไทย | native input ลด error |
| อีเมล (Q6, text) | แป้นอีเมล (เห็น @), validate inline, micro-copy "ส่งผลประเมิน + เอกสาร — ไม่สแปม" | reassurance ตรงจุด; ตอนนี้ "ขอไปทำไม" มีคำตอบชัด |
| ช่องทางติดต่อกลับ (Q36, radio) | **ปุ่มใหญ่ เน้น LINE** (สีเขียว + โลโก้ LINE เด่น) | LINE คือช่องทางหลักของธุรกิจ |
| ช่วงเวลาติดต่อกลับ (Q37, radio 3) | **ปุ่มช่วงเวลา** สไตล์แอปจองคิว (เช้า/บ่าย/เย็น) | visual เลือกง่าย |
| รู้จักจากช่องไหน (Q7, dropdown 6) | **ชิป/ไอคอนโลโก้** เลือกอันเดียว (FB/IG/TikTok/Google...) | คำถามการตลาด เบาสุด — วางก่อน submit ไม่รบกวน assessment |
| รับรองว่าข้อมูลจริง (certification) | **checkbox attestation** ติดกับปุ่ม submit | รับรองหลังกรอกครบ = ตำแหน่งตามธรรมชาติ |

→ ปุ่ม **"ส่งแบบประเมิน →"** (อาจแบ่งเป็น 1–2 หน้าจอ: ติดต่อ+นัดเวลา / แล้วค่อยรู้จักจากช่องไหน+รับรอง)

## 5. หน้า Welcome & Confirmation (จุด peak-end)

**[0] Welcome (ก่อน consent):**
- พาดหัวคุณค่า: "เช็คโอกาสผ่านวีซ่าฟรี ใน 2 นาที" + ทีมจะเตรียมคำแนะนำเฉพาะคุณ
- Trust signal: จำนวนเคส / อัตราผ่าน (ถ้ามีจริง) — วางตรงนี้ ไม่ใช่ footer
- Privacy 1 บรรทัด: "ข้อมูลของคุณปลอดภัยและเป็นความลับ"
- ปุ่มเดียว: "เริ่มเลย →" (active, challenge-framed)

**[8] Confirmation (หลัง submit):**
- ขอบคุณ + สรุปสั้นว่าจะเกิดอะไรต่อ ("ทีมวีซ่าจะติดต่อกลับภายใน X ชั่วโมง")
- ปุ่ม **Add LINE @itinerry** เด่นๆ (ปิดวงจรไปช่องทางหลัก)
- บอกขั้นตอนถัดไปสั้นๆ → จบด้วยความรู้สึกบวก (peak-end)

## 6. Anti-patterns ที่ต้องเลี่ยง (เฉพาะฟอร์มนี้)

- ❌ โชว์ **% progress** — กระตุกเพราะ branching; ใช้ขั้น/section แทน
- ❌ เปิดด้วย **กำแพง PDPA** — first impression แย่สุดสำหรับคนกังวล
- ❌ **text ว่าง** สำหรับวันที่/ประเทศ — ใช้ picker/searchable เสมอ
- ❌ ทำให้คำถาม sensitive **เหมือนการสอบสวน** — ต้องมี reassurance
- ❌ วาง **ทุกคำถามใน scroll เดียว** — นั่นคือ Google Form เดิม ไม่ได้แก้อะไร
- ❌ ใส่ **ลูกเล่น/gamification หนักๆ** — งานวิจัยพบว่าเพิ่มความพึงพอใจแต่บางครั้งลดคุณภาพข้อมูล และเรื่องวีซ่าต้องการข้อมูลแม่นยำ

## สรุปลำดับความสำคัญในการลงมือ

1. **ทีละคำถาม + branching ที่ซ่อนคำถามไม่เกี่ยว** (ผลกระทบสูงสุด)
2. **Reassurance ที่คำถาม sensitive + กรอบเชิงบวกที่คำถามสัญญาณบวก** (หัวใจของบริบทวีซ่า)
3. **Native input ทุกจุด (date/tel/email/search)** (แก้ pain mobile ทันที)
4. **Welcome + Confirmation screen** (peak-end)
5. **Progress แบบ section + help text ทุกศัพท์เฉพาะ**
