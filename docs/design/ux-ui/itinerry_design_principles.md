# itinerry · Design Principles
### หลักการออกแบบสำหรับ Visa Assessment (และงาน high-anxiety อื่นๆ)

> **กฎเดียวที่อยู่เหนือทุกข้อ:** งานนี้คนใช้กำลัง *กังวล* และ *ไม่รู้เรื่องวีซ่า* — หน้าที่ #1 ของดีไซน์คือทำให้เขา **กล้าตอบตามจริงโดยไม่รู้สึกถูกตัดสิน** และ **รู้สึกว่าจัดการได้** สิ่งที่ดูเท่ที่สุดไม่ใช่สิ่งที่เวิร์กที่สุด — สิ่งที่ลด friction ได้มากที่สุดต่างหากที่เวิร์ก

---

## เลนส์ที่มองทุกการตัดสินใจ (The Lens)

| แกน | itinerry |
|---|---|
| **ความวิตกกังวล** | **สูง** — วีซ่าผิดพลาดไม่ได้ |
| **ความรู้ในเรื่อง** | **ต่ำ** — ส่วนใหญ่ทำครั้งแรก |
| **แพลตฟอร์ม** | **Mobile-first**, LINE เป็นช่องทางหลัก |
| **ผู้ใช้** | คนไทย — ภาษาเป็นกันเอง ไม่ราชการ |

ทุกหลักการด้านล่างคือการตอบสนองต่อเลนส์นี้

---

## หลักการ 9 ข้อ

### 1 · ลด friction ไม่ใช่เพิ่มลูกเล่น
"ไม่น่าเบื่อ" = เอาภาระทางความคิดออก ไม่ใช่ใส่เกม
- **ทำไม:** ฟอร์มทีละคำถามแบบสนทนาเพิ่ม completion ~25–40% (หลักฐานแน่น) แต่ gamification หนักๆ เพิ่มความพึงพอใจกลับ *บางครั้งลดคุณภาพข้อมูล* (หลักฐานก้ำกึ่ง) — และวีซ่าต้องการข้อมูลแม่นยำ
- **ในงานนี้:** ทีละคำถาม + auto-advance, ไม่มีแต้ม/badge/leaderboard

### 2 · กล้าตอบจริง สำคัญกว่าการสร้างภาพ *(หัวใจของบริบทวีซ่า)*
คำถามที่คนกลัว → ให้กำลังใจ; คำถามที่เป็นสัญญาณบวก → กรอบให้ภูมิใจ
- **ทำไม:** บริการต้องใช้ข้อมูลจริง แต่คนกลัวว่ายอมรับแล้วเสียโอกาส
- **ในงานนี้:** "เราไม่ตัดสิน" ที่คำถามถูกปฏิเสธ/overstay · "ยิ่งมากยิ่งดีต่อเคส" ที่ประวัติวีซ่า/ความผูกพันไทย · privacy note ติดกับคำถามการเงิน (ไม่ใช่ซ่อนใน consent)

### 3 · ทำให้รู้สึกสั้น
- **ทำไม:** *ความซับซ้อนสำคัญกว่าความยาว* และคนหลุดมากที่ "ช่วงกลางยืดเยื้อ"; เส้นทางที่ปรับเฉพาะคนรู้สึกสั้นและตรงกว่า
- **ในงานนี้:** branching ซ่อนคำถามที่ไม่เกี่ยวไปเลย (เห็นเฉพาะ ~15–20 ข้อของตัวเอง) · 1 การตัดสินใจต่อหน้าจอ · สัญญา ~2 นาที

### 4 · Progress ให้กำลังใจ ไม่ใช่ขู่
- **ทำไม:** progress bar *ทำงานย้อนกลับ* เมื่อช่วงต้นรู้สึกช้า (meta-analysis 32 การทดลอง) และการโชว์ % ที่ด้านบนเคยเพิ่ม drop-off (ข้อมูล SurveyMonkey)
- **ในงานนี้:** progress แบบ "ขั้นที่ X จาก 3" ไม่ใช่ % (เพราะ branching ทำให้ % กระตุก) · ทำช่วงต้นให้รู้สึกเร็ว · ไม่โชว์ "37 ข้อ" ให้คนตกใจ

### 5 · Commitment มาจากการลงแรง ไม่ใช่การรีดข้อมูล
- **ทำไม:** พอคนลงแรงทำ assessment แล้วจะอยาก "รับผล" (sunk-cost / IKEA effect) — ขอข้อมูลตอนนั้นได้ผลและไม่รู้สึกเป็นกับดักเก็บ lead
- **ในงานนี้:** ย้าย ชื่อ/เบอร์/อีเมล ไปท้าย กรอบเป็น "เกือบเสร็จ! บอกเราว่าจะส่งผลให้คุณยังไง"
- **tradeoff ที่รับไว้:** ถ้าคนหลุดตอนกรอกติดต่อจะตามไม่ได้ → กันด้วย transition ที่กรอบเป็นการส่งมอบคุณค่า + autosave draft

### 6 · Mobile-native, thumb-first
- **ในงานนี้:** native input ทุกจุด (date / tel / email / searchable country) · คอลัมน์เดียว · tap target ใหญ่ · ปุ่มหลัก sticky ด้านล่างในโซนนิ้วโป้ง

### 7 · ภาษาง่าย + ช่วยตรงจุด (just-in-time help)
- **ทำไม:** knowledge ต่ำ ต้องอธิบายศัพท์เฉพาะทุกตัว; *processing fluency* — อะไรเข้าใจง่ายจะถูกมองว่าน่าเชื่อถือกว่า
- **ในงานนี้:** helper บนการ์ดเลือกประเภทวีซ่า/อาชีพ · อธิบาย Invitation Letter / DBD / "ความผูกพันกับไทย" ตรงจุดที่ถาม

### 8 · เปิดอุ่น–ปิดสวย (Peak-End Rule)
- **ทำไม:** ความรู้สึกตอน *เปิด* และ *ปิด* กำหนดความทรงจำต่อทั้งแบรนด์ และหน้าแรกคือจุดที่คนตัดสินใจว่าจะไปต่อไหม
- **ในงานนี้:** เปิดด้วย welcome (คุณค่า + ความน่าเชื่อถือ + "2 นาที") *ไม่ใช่* กำแพง PDPA · ปิดด้วย confirmation ที่บอกขั้นตอนถัดไป + ปุ่ม Add LINE

### 9 · การรอถูกออกแบบ ไม่ใช่ปล่อยให้สะดุด
- **ทำไม:** การรอที่ *อธิบายได้ + มีอะไรเกิดขึ้น* รู้สึกสั้นกว่าเสมอ (Maister)
- **ในงานนี้:** prefetch คำถามชุดถัดไปทันทีที่เลือกจุดแยก + optimistic transition + **ช้างน้อยจดโน้ตบนไอแพด** ใน 3 จังหวะ fetch → เปลี่ยน "กำลังโหลด" เป็น "ทีมกำลังจดคำตอบของคุณ"

## ระบบสี / ฟอนต์ (ใช้ให้สม่ำเสมอ)

- **ฟอนต์:** Noto Sans Thai (เนื้อหาไทย) + Plus Jakarta Sans (UI/ตัวเลข/ป้าย)
- **สีหลัก:** navy `#1b3d5c` (หัวข้อ) · sky `#44a8db` (interactive, step, active)
- **Yellow `#ffd166` = ปุ่ม CTA เปลี่ยนใจ *เท่านั้น*** — อย่าใช้พร่ำเพรื่อ ไม่งั้นหมดพลัง
- **LINE green `#06C755` = ปุ่ม LINE เท่านั้น**
- **Radius:** การ์ด 1rem · container 1.5rem · ปุ่ม/ชิป/step = pill
- สีฟ้าอ่อน (`accent-bg`) = โทน reassurance · เขียว (`success-bg`) = โทนเชิงบวก · แดง/อำพัน = flag เท่าที่จำเป็น (อย่าทำให้คำถาม sensitive ดูเหมือนสอบตก)

## Anti-patterns ที่ต้องเลี่ยง

- ❌ โชว์ **% progress** (กระตุกเพราะ branching) → ใช้ขั้น/section
- ❌ เปิดด้วย **กำแพง PDPA** → welcome อุ่นๆ ก่อน
- ❌ **text ว่าง** ให้พิมพ์วันที่/ประเทศ → picker/searchable เสมอ
- ❌ ทำคำถาม sensitive **เหมือนการสอบสวน** → ต้องมี reassurance
- ❌ วาง **ทุกคำถามใน scroll เดียว** → นั่นคือ Google Form เดิม
- ❌ **gamification หนักๆ** เพื่อความสนุก → เสี่ยงคุณภาพข้อมูล
- ❌ **เก็บ lead ก่อนให้คุณค่า** → ให้ทำ assessment ก่อน ขอติดต่อทีหลัง

---

## หมายเหตุเรื่องน้ำหนักหลักฐาน (ซื่อสัตย์กับทีม)

- **แน่น/ทำซ้ำได้:** ลด cognitive load, ทีละคำถาม, complexity > length, peak-end, processing fluency
- **ก้ำกึ่ง/ขึ้นกับบริบท:** ผลของ gamification, ผลของ progress bar (ดีหรือร้ายขึ้นกับว่าช่วงต้นรู้สึกเร็วไหม), การถามอีเมลก่อน vs หลัง
- หลักการกลุ่มหลังให้ตัดสินจากบริบท itinerry (high-anxiety + assessment ยาว) ไม่ใช่ยึดตัวเลขจาก survey สั้นๆ
